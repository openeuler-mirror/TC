议题9：openEuler 22.03 LTS ARM64 版本(5.10 内核）内核页大小及支持CPU规格决策：  - 谢秀奇
  决策点1：openEuler 22.03 LTS 是否默认采用 4K 页表 （配合 48 BIT VA/PA)
  投票结果：反对 0，弃权 0， 赞成 16 
  决策点2：openEuler 22.03 LTS 如果默认采用 4K 页表，是否需要在单独出 64K 页表的 kernel 包。
  投票结果：反对 0，弃权 1，赞成 15
  
  内核默认页大小的切换对社区的兼容性有重大的影响，需要考虑构建环境的切换，南向驱动的兼容，北向应用的兼容，虚拟化场景下新旧版本的共存等等。
  openEuler社区技术委员会需要针对这个技术决策发布白皮书，说明切换的影响以及各个场景下的应对策略指南。
  白皮书的开发由 胡欣蔚 和 谢秀奇 协调，各个 SIG 支持，目标在21年12月31日前在社区发布。
  决策点3：openEuler 22.03 LTS 支持最大 CPU 数：NR_CPUS=4096, NODES 支持 128. (本次会议不决策，sig 自己讨论即可）

 （20.03 原版本配置：64K, 48 BIT VA/PA, NR_CPUS 1024, NODES 16)

  之前的讨论链接：
  https://gitee.com/openeuler/kernel/issues/I4HDHZ

  理由1：从讨论看，64K 兼容性问题比较多，且业界OS多采用4K，方便驱动及上层软件在兼容性认证，
         也利于支撑更多的使用场景。性能上，64K 优势场景，可以采用大页，或者单独 64K 版本支持。
  理由2：52 BIT 大内存支持，需要 64K 页，4K 页最大支持 48 BIT VA/PA, 256T 内存空间。如果要支
         持更大 PB 级内存，需要 64K。
  理由3：从未来几年诉求，和业界OS支持规格来看，CPU 数目增加是趋势，从兼容性和规格考虑，
         NR_CPUS 与业界OS最大的 4096 一致。当前4路飞腾S2500 最大 32 个 nodes，为满足未来几年
         的演进诉求，NODES 数拟最大支持 128.

议题6： OBS 优化后的 Project Config 应用到 Mainline 工程 - 华为 陈燕潘
openEuler 构建优化分析发现 OBS Project Config 存在冗余配置，导致软件包每次构建需要预装大量 rpm，其中一些 rpm 并非每个软件包构建所需。构建过程安装大量冗余的 rpm，导致设备 IO 成为构建的性能瓶颈，增加了软件包的构建时长。针对这个问题需要对 Project Config 进行精简，仅预装每个软件包共同需要的 rpm，其它需要的 rpm 由各自软件包 spec 的 BuildRequires 写明。前期试验，该项可达到的优化效果：平均每个软件包构建时少安装的 rpm 80+个（21%），缩短构建时长 25秒（20%）。
现计划将精简后的 Project Config 应用到 Mainline 工程，由于减少了预装的 rpm，会导致 2 类问题：
1. 某些包缺少必备的构建依赖导致构建失败
2. 少预装某些 rpm，软件包可以构建成功，但缺失某些功能
第1类问题结果是构建失败，比较容易识别，从日志也可以分析缺少的构建依赖并补齐。前期在试验工程中并识别了 300+ 个问题，已经修改闭环。由于 Mainline 工程的代码在持续合入，应用该项优化后，预计会有部分软件包存在该类问题，预计在 30 个以内。第2类问题比较隐蔽，需要对比优化前后 rpm 内容的差异。计划是保存当前已构建的 rpm，将该项优化应用后构建新的 rpm，与之前保存的 rpm 对比，识别有差异的软件包，补齐构建依赖解决。当前正在开发工具，通过 ABI 及 provides 等信息识别 rpm 的差异，计划本周能开发完成工具。
本次议题是介绍上述计划，实施该项优化时会通过邮件方式通知社区。

结论：
1、当前精简后的 Project Config 应用在 https://build.openeuler.org/project/prjconf/openEuler-Mainline-copy 工程，如有小伙伴需要可获取试用。
2、对比 openEuler-Mainline-copy 构建的 rpm 与当前 Mainline 构建的 rpm 是否存在接口层面的差异，根据对比结果 TC 决策是否应用到 Mainline 工程。对比功能目前正在开发，预计一周内可以生成对比结果，争取在下周 TC 增补例会上评审，如果时间赶不上再申请单独会议评审。

议题3： openEuler OBS 构建 Project Config 及 spec 优化                    chenyanpan               30分钟
内容：OBS 每次构建软件包会创建新的构建环境（build environment），在构建环境中 preinstall 一些必备软件包，然后 install 这些必备软件包及其依赖，以及 spec 中指定的 BuildRequires 及其依赖。当前 preinstall 安装多余软件包，部分软件包 spec 存在多余依赖，导致每次创建构建环境至少需要安装 300+ 个 rpm，占用空间 1.2GB，耗时 60+ 秒。需要对 Project Config 及 spec 进行优化，减少构建过程安装的 rpm 数量，避免 IO 资源浪费，缩短构建时间。
同时去掉软件包 spec 多余依赖，避免安装时间、空间浪费，也避免安装多余 rpm 引入的安全风险。
结论：TC 同意优化方案落地实施，具体步骤如下
1. 先在 Mainline 工程上应用，chenyanpan 提供 Mainline 工程中因 spec 缺少依赖导致构建失败的软件包列表(约300个)，软件包责任人整改 spec 补充依赖；整改后再将优化的 Project Config 应用到 Mainline 工程。整改过程责任人可以先 branch openEuler-Mainline-copy 到个人工程进行开发验证。有些软件包可能由于缺少依赖包，构建结果为 succeeded 但功能存在缺陷，这类风险的软件包无法提前识别并规避，只能发现问题再解决。
2. chenyanpan 提供依赖不合理的关键软件包列表，包括 rpm-build、pam、shadow 等，软件包责任人进行整改。
3. 同时研究 OBS 复用构建环境方法（前期调研 OBS 不支持复用机制，可能需要通过修改 OBS 才能支持）。

议题5： 移交autotrace、低版本jQuery和firefox至recycle或者永久备案不修复CVE           夏森林      15分钟
背景：当前这几个包的CVE相关的issue日益增多，但是由于各自的原因，社区maintainer无法逐一修复：
（1）  js-jquery1和js-jquery2：涉及CVE 3个，社区发布的大部分CVE的补丁是针对js-jquery3的，并不适用低版本，代码差距大，适配难度高
（2）  firefox：涉及CVE 116个，其中7分以上55个，firefox79及后续版本引入了fdk-aac-free包，其license 为FDK-AAC。License 地址：https://fedoraproject.org/wiki/Licensing/FDK-AAC（网址中第3个license----NO PATENT LICENSE没有专利授权）；
（3）  autotrace：涉及CVE 54个，由于社区无人维护，无法修复；
结论：
（1）同意不修复js-jquery1和js-jquery2的CVE，评估在SP2版本使用js-jquery3的替代方案的风险，建议在SP2中移除js-jquery1和js-jquery2。
（2）同意暂不修复firefox的CVE，夏森林分析剥离fdk-aac-free的方案，可以通过剥离风险包的方式解决license风险问题，同时考虑替代方案，如使用chromium

3）依赖包停止维护的软件包如何操作更为合理。
概述：类似deepin-manual软件包由于python2不再维护导致其不能成功编译，需将其移入sig-recycle。针对此类场景如何操作更为合理？
背景：https://gitee.com/openeuler/community/pulls/1590
   openEuler社区当前软件包退出原则详见： https://gitee.com/openeuler/community/blob/master/zh/technical-committee/governance/software-management.md
结论：
整体按照已经发布的原则来处理：
由于软件的依赖包在上游社区停维或者不活跃，并且
1、社区sig没有意愿继续维护的软件包统一放入recycle，并且从社区版本发布中删除
2、社区sig有意愿继续维护，则不做移入recycle的操作，社区对应sig推动快速解决依赖包带来的影响；解决后及时合入到社区版本(未来发布和已发布的LTS&SP)中
case by case看这个PR：https://gitee.com/openeuler/community/pulls/1621
这个软件包不带入21.03版本，其它后续版本视上游社区问题解决情况来定；
但是需要统信的DDE-sig组将此包合入到统信的版本中
